# 差旅报销单接口文档

版本：V1.0  
来源：`差旅报销单概要设计.docx`，修订日期 2026-01-26  
适用模块：差旅报销单列表、报销单详情、补录行程、补助信息、补助日历、费用合计、基础数据

## 1. 通用约定

### 1.1 基础路径

```text
/api/v1
```

### 1.2 请求与响应格式

请求体和响应体均使用 `application/json; charset=utf-8`。

统一响应结构：

```json
{
  "code": "0",
  "message": "success",
  "data": {},
  "traceId": "202605221130000001"
}
```

分页响应结构：

```json
{
  "code": "0",
  "message": "success",
  "data": {
    "pageNo": 1,
    "pageSize": 20,
    "total": 100,
    "records": []
  }
}
```

### 1.3 通用错误码

| code | 说明 |
| --- | --- |
| 0 | 成功 |
| 400 | 请求参数错误 |
| 401 | 未登录或登录已失效 |
| 403 | 无操作权限 |
| 404 | 数据不存在 |
| 409 | 数据冲突，例如行程人员和日期重复 |
| 422 | 业务校验失败 |
| 500 | 系统异常 |

### 1.4 单据状态

| 状态值 | 状态名称 | 说明 |
| --- | --- | --- |
| DRAFT | 未提交 | 保存报销单、撤回后状态 |
| APPROVING | 审批中 | 提交后状态 |
| APPROVED | 审批通过 | 审批通过后状态 |
| COMPLETED | 已完成 | 放款后状态 |
| VOIDED | 已作废 | 作废后状态 |

状态流转：

| 操作 | 原状态 | 目标状态 |
| --- | --- | --- |
| 保存报销单 | 无、未提交 | 未提交 |
| 提交 | 未提交 | 审批中 |
| 审批通过 | 审批中 | 审批通过 |
| 放款 | 审批通过 | 已完成 |
| 撤回 | 审批中 | 未提交 |
| 编辑 | 未提交 | 未提交 |
| 删除 | 未提交 | 无 |
| 作废 | 未提交、审批中、审批通过 | 已作废 |


### 3.5 删除报销单

```http
DELETE /api/v1/travel-reimbursements/{reimbursementId}
```

说明：仅 `DRAFT` 状态允许删除。

### 3.6 提交报销单

```http
POST /api/v1/travel-reimbursements/{reimbursementId}/submit
```

提交时后台必须执行以下校验：

| 校验项 | 规则 |
| --- | --- |
| 基本信息必填 | 报销标题、出差事由、报销人、报销部门、费用归属公司、业务类型均不能为空 |
| 标题和事由长度 | 均不能超过 500 字 |
| 备注长度 | 不能超过 1000 字 |
| 补录行程必填 | 每条行程的出行人、出发城市、到达城市、出发日期、到达日期、行程说明均不能为空 |
| 日期范围 | 到达日期不能早于出发日期，且不能晚于当前日期 |
| 行程重复 | 同一出行人的日期范围不能重叠 |
| 补助金额 | 只能为正数，且不能大于对应标准金额 |

响应：

```json
{
  "code": "0",
  "message": "提交成功",
  "data": {
    "reimbursementId": "RB202605220001",
    "status": "APPROVING",
    "statusName": "审批中"
  }
}
```

### 3.7 撤回报销单

```http
POST /api/v1/travel-reimbursements/{reimbursementId}/withdraw
```

说明：`APPROVING` 状态允许撤回，撤回后状态为 `DRAFT`。

### 3.8 作废报销单

```http
POST /api/v1/travel-reimbursements/{reimbursementId}/void
```

请求体：

```json
{
  "reason": "单据填写错误"
}
```

### 3.9 审批通过

```http
POST /api/v1/travel-reimbursements/{reimbursementId}/approve
```

说明：审批流系统调用或有审批权限的用户调用，审批通过后状态为 `APPROVED`。

### 3.10 放款完成

```http
POST /api/v1/travel-reimbursements/{reimbursementId}/disburse
```

请求体：

```json
{
  "paymentTime": "2026-05-22 15:30:00",
  "paymentNo": "PAY202605220001"
}
```

说明：`APPROVED` 状态允许放款，放款后状态为 `COMPLETED`。

## 4. 补录行程接口

### 4.1 新增补录行程

```http
POST /api/v1/travel-reimbursements/{reimbursementId}/trips
```

请求体：

```json
{
  "travelerId": "13AB3A3F72409002",
  "departureCityNo": "10458",
  "arrivalCityNo": "10119",
  "departureDate": "2026-05-18",
  "arrivalDate": "2026-05-20",
  "description": "武汉到北京项目实施"
}
```

字段规则：

| 字段 | 类型 | 必填 | 规则 |
| --- | --- | --- | --- |
| travelerId | string | 是 | 出行人 |
| departureCityNo | string | 是 | 出发城市 |
| arrivalCityNo | string | 是 | 到达城市 |
| departureDate | string | 是 | 出发日期 |
| arrivalDate | string | 是 | 到达日期，不能早于出发日期，不能晚于当前日期 |
| description | string | 是 | 行程说明，最多 500 字 |

业务规则：

1. 仅可补录未从申请单带入或未产生费用的行程信息。
2. 同一出行人的行程日期范围不能重复或重叠。
3. 保存补录行程后，后台需要自动生成一条补助信息和对应补助日历。
4. 跨天跨城行程的补助城市按到达城市匹配。

### 4.2 更新补录行程

```http
PUT /api/v1/travel-reimbursements/{reimbursementId}/trips/{tripId}
```

说明：更新后需要重新计算对应补助信息和补助日历。

### 4.3 删除补录行程

```http
DELETE /api/v1/travel-reimbursements/{reimbursementId}/trips/{tripId}
```

说明：删除行程时，需要同步删除该行程关联的补助信息和补助日历。

### 4.4 复制补录行程

```http
POST /api/v1/travel-reimbursements/{reimbursementId}/trips/{tripId}/copy
```

请求体：

```json
{
  "departureDate": "2026-05-21",
  "arrivalDate": "2026-05-22"
}
```

说明：复制时默认带出原行程数据；保存时生成新行程，仍需执行人员和日期重复校验。


```json
{
  "items": [
    {
      "calendarId": "CAL202605180001",
      "meal": {
        "checked": true,
        "amount": 100
      },
      "traffic": {
        "checked": false,
        "amount": 0
      },
```

后台校验：

| 校验项 | 规则 |
| --- | --- |

| 金额上限 | 金额不能大于对应标准金额 |
| 标准金额 | 以城市类型和补助类型重新计算，不能信任前端传入的标准金额 |

## 8. 补助标准

| 补助类型 | 城市类型 | 标准 |
| --- | --- | --- |
| 餐费补助 | 一线城市 | 100 元/天 |
| 餐费补助 | 二线城市 | 80 元/天 |
| 餐费补助 | 三线城市 | 50 元/天 |
| 交通补助 | 全部城市 | 40 元/天 |
| 通讯补助 | 全部城市 | 40 元/天 |

城市类型来源于城市基础数据 `cityType`：

| cityType | 说明 |
| --- | --- |
| 1 | 一线城市 |
| 2 | 二线城市 |
| 3 | 三线城市 |

## 9. 后台校验规则汇总

| 场景 | 校验规则 |
| --- | --- |
| 保存报销单 | 标题、事由最长 500 字；备注最长 1000 字 |
| 提交报销单 | 基本信息必填、行程必填、行程日期不重复、补助金额合法 |
| 新增或更新行程 | 出行人、城市、日期、说明必填；到达日期不能早于出发日期；到达日期不能晚于当前日期 |
| 行程重复校验 | 同一出行人任意两条行程日期范围不能重叠 |
| 删除行程 | 同步删除关联补助信息和补助日历 |
| 保存补助日历 | 未选中项目金额为 0；选中项目金额大于 0 且不大于标准金额 |
| 金额计算 | 标准金额由后台按城市类型和补助类型计算，前端传值仅作展示参考 |

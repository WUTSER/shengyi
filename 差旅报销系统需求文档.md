## 1. 引言

### 1.1 编写目的

本文档在《差旅报销单概要设计》基础上，将业务背景、功能清单、页面逻辑、数据规则与校验要求整理为可指导产品设计、研发实现与测试验收的需求说明，供项目成员统一理解系统边界与验收标准。

### 1.2 业务背景

员工因公出差或为企业办理业务时，往往先自行垫付费用，业务结束后向企业报账结算；也可能先向公司借款办理业务，办结后与公司对账，多退少补。若出差全程通过线上预订并由公司、服务商或供应商垫付，员工返程后仅需凭票据报账，此时报销金额可能为 0，由企业财务与服务商或供应商对账结算。

当前报销场景存在以下痛点：

| 痛点类别 | 说明 |
| --- | --- |
| 审批耗时耗力 | 领导侧重费用真实性；财务需综合审核时间、金额、地区等，人力与时间成本高 |
| 业务真实性难保证 | 需核验行程时间是否重复、是否与发票时间吻合；消费金额是否与发票一致；是否存在与行程无关的发票 |
| 票据合规性 | 查真查重、抬头税号一致性、连号检测、有效期等难以靠人工全覆盖 |
| 财务与员工矛盾 | 审批慢、打回多；票据签收状态不清晰，易产生争议 |

本系统首期聚焦**差旅报销单的填报、行程补录、补助计算与费用分摊**，为后续对接审批流、票据合规等能力预留扩展空间。

### 1.3 建设目标

1. 提供报销单列表查询与详情维护能力，支撑员工自助填报与查询。
2. 支持补录未从申请单带入或未产生费用的行程，并自动生成补助信息与补助日历。
3. 按城市类型与补助类型自动匹配标准金额，控制补助金额不超过标准。
4. 支持费用按公司与项目进行分摊，分摊比例与金额联动校验。
5. 前后端一致的必填、重复、金额与分摊校验，降低疏漏与返工。

### 1.4 系统范围

**本期包含：**

- 报销单列表（查询、展示、跳转详情）
- 报销单详情页（头部、基础信息、补录行程、补助信息、费用合计、费用归属及分摊、备注、底部操作）
- 基础数据（费用归属公司、报销部门、员工、业务类型树、城市、项目）
- 单据状态：草稿、已完成（提交后）、已作废

**本期不包含（概要设计未定义或仅作背景描述）：**

- 发票/票据明细录入与合规性自动审核
- 完整多级审批流界面（可与统一审批中心后续对接）
- 差旅申请单带入行程的对接（仅提示“仅可补录未从申请单带入…”）

### 1.5 术语与缩写

| 术语 | 说明 |
| --- | --- |
| 报销单 | 员工差旅费用报账的主单据 |
| 补录行程 | 手工补充的出差行程，非申请单带入且无关联费用 |
| 补助信息 | 由补录行程关联生成的出差补助汇总记录 |
| 补助日历 | 按出差日期逐日展示的餐费、交通、通讯补助明细 |
| 费用分摊 | 将补助总金额按费用归属公司与项目拆分比例与金额 |

## 3. 业务流程

### 3.1 报销业务简述

```mermaid
flowchart LR
  A[员工出差/垫资] --> B[填写报销单]
  B --> C[补录行程]
  C --> D[生成补助信息与日历]
  D --> E[调整补助金额]
  E --> F[费用合计与分摊]
  F --> G[提交]
  G --> H[财务/领导处理]
```

### 3.2 单据状态

| 状态编码 | 状态名称 | 说明 |
| --- | --- | --- |
| 0 | 草稿 | 表单未填完或已保存未提交 |
| 1 | 已完成 | 提交成功，表单数据完整且校验通过 |
| 2 | 已作废 | 作废后的单据 |

### 3.3 状态流转

| 操作 | 原状态 | 目标状态 | 说明 |
| --- | --- | --- | --- |
| 新增/保存 | — | 草稿(0) | 数据未填完或未提交 |
| 提交 | 草稿(0) | 已完成(1) | 全部必填与业务校验通过 |
| 作废 | 草稿(0)、已完成(1) | 已作废(2) | 按业务规则允许作废 |

> **说明：** 概要设计状态表为「草稿 / 已完成 / 已作废」三态。若项目需对接审批中、审批通过、放款完成等状态，应在迭代需求中单独补充状态机与接口，避免与本文档三态混用。

## 5. 业务规则与校验需求

### 5.1 通用校验原则

- 概要设计明确要求：**页面与后台接口均需实现相同校验逻辑**，以后台校验为最终依据。
- 标准金额由后台按城市类型与补助类型计算，**不得信任前端传入的标准金额**。

### 5.2 保存/编辑报销单

| 校验项 | 规则 |
| --- | --- |
| 报销标题 | 必填，≤ 500 字 |
| 出差事由 | 必填，≤ 500 字 |
| 报销人/部门/费用归属公司/业务类型 | 必填 |
| 备注 | 选填，≤ 1000 字 |

### 5.3 补录行程

| 校验项 | 规则 |
| --- | --- |
| 必填项 | 出行人、出发/到达城市、出发/到达日期、行程说明 |
| 日期逻辑 | 到达日期 ≥ 出发日期；到达日期 ≤ 当前日期 |
| 行程说明 | ≤ 500 字 |
| 重复校验 | 同一出行人任意两条行程日期范围不得重叠 |

### 5.4 补助日历

| 校验项 | 规则 |
| --- | --- |
| 未勾选补助项 | 金额必须为 0 |
| 已勾选补助项 | 金额 > 0 且 ≤ 对应标准金额 |
| 补助金额汇总 | 补助信息中补助金额只能为正数，且不大于标准总额（提交时） |

### 5.5 提交报销单

提交时需校验：

1. 基础信息全部必填项已填且长度合规。  
2. 至少存在合规的补录行程（每条行程字段完整、日期合法、人员日期不重复）。  
3. 补助金额合法（正数且不超过标准）。  
4. **费用分摊：分摊比例合计为 100%；分摊金额合计等于费用合计中的补助总金额。**

### 5.6 删除行程的级联规则

删除补录行程时，必须同步删除该行程关联的补助信息及补助日历，并重新计算费用合计与分摊展示（若实现自动重算）。

| 业务类型 | businessTypeId, businessTypeNo, businessTypeName, thereSubordinateNode, superiorId | 树形选择；`thereSubordinateNode`：1 有下级，0 无；`superiorId` 顶级为 `none` |
| 城市 | cityNo, cityName, cityType | 行程城市；cityType：1 一线、2 二线、3 三线 |
| 项目 | projectId, projectNo, projectName | 费用分摊 |

### 6.2 业务单据实体（逻辑模型）

| 实体 | 说明 | 关键关联 |
| --- | --- | --- |
| 差旅报销单 | 主单 | 1:N 行程、补助、分摊 |
| 补录行程 | 出差段 | N:1 报销单；1:1 补助信息（生成） |
| 补助信息 | 行程维度补助汇总 | 1:N 补助日历 |
| 补助日历 | 按日补助明细 | 含餐费/交通/通讯勾选与金额 |
| 费用分摊行 | 归属与比例 | N:1 报销单 |

### 6.3 示例主数据

概要设计附录提供了费用归属公司、报销部门、员工、业务类型、城市、项目等 JSON 样例，实现阶段应通过主数据服务或配置中心维护，需求阶段以「字段结构一致、编码可映射」为验收条件。完整样例见概要设计 **5.3 页面数据** 章节。

## 10. 验收标准（摘要）

| 编号 | 验收场景 | 通过条件 |
| --- | --- | --- |
| AC-01 | 列表查询 | 各查询条件生效，结果字段与展示要求一致 |
| AC-02 | 新建草稿 | 必填校验生效，保存为草稿状态 |
| AC-03 | 补录行程 | 保存后列表展示，生成补助信息与日历；重复日期被拒绝 |
| AC-04 | 跨城补助 | 多日行程按到达城市类型匹配餐补标准 |
| AC-05 | 补助日历 | 勾选/全选逻辑正确，金额不超标准，未勾选为 0 |
| AC-06 | 费用合计 | 与补助信息汇总一致 |
| AC-07 | 费用分摊 | 均摊、联动首行、比例 100%、金额等于补助总金额 |
| AC-08 | 提交 | 全部校验通过变为已完成；任一校验失败有明确提示 |
| AC-09 | 删除行程 | 关联补助数据同步删除 |
| AC-10 | 作废 | 允许状态下变为已作废 |
